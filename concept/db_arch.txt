3. แผนผังความสัมพันธ์ (Logical ER Diagram Concept) 
1.​ Identity & Group Flow: Tenants → Users → Groups ↔ Group Members 
○​ นัยสำคัญ: หัวหน้ากลุ่มสามารถเข้าถึง daily_aggregates และ weekly_aggregates ของกลุ่มตน
เองได้ 
2.​ Reporting Flow: Trade History → Worker Process → Aggregates (Daily/Weekly/Monthly) 
○​ นัยสำคัญ: มีระบบ Background Worker (เช่น Go routine หรือ Cron) คอยคำนวณกำไรสะสมและ 
Win Rate แล้วบันทึกลงตาราง Aggregates ทุกสิ้นวัน/สัปดาห์/เดือน 
3.​ Dashboard Flow: User Login → Dashboard Widgets → Aggregates Data 
○​ นัยสำคัญ: หน้า Dashboard จะดึงข้อมูลจากตารางสรุปผลทันที ทำให้ประหยัดทรัพยากรฐานข้อมูล
มหาศาล 
4. การออกแบบเพื่อรองรับ Scaling และ Performance 
●​ Materialized Views / Aggregated Tables: ใช้ตารางสรุปผลแทนการ Query trade_history โดย
ตรงสำหรับหน้า Dashboard 
●​ Data Retention Policy: ข้อมูลใน audit_logs และ bot_events ที่เก่ากว่า 6 เดือนจะถูก Archive ไป
ยัง Cold Storage เพื่อรักษาประสิทธิภาพ DB 
5. ความปลอดภัยและการกำหนดสิทธิ์ (Security Focus) 
●​ Report Access Control: การสั่ง Export รายงานต้องตรวจสอบสิทธิ์ report.view ผ่าน Middleware 
เสมอ
